# v0.5 Development Plan

This document outlines the architectural changes, technical breakdown, and testing strategy to address the top 5 deficiencies identified in the `Critique.md` analysis, bringing the engine closer to its performance, structural, and feature goals.

## 1. Deficiencies & Technical Breakdown

### 1.1 ECS Misnomer (Data-Oriented ECS Refactor)
**The Problem:** The current implementation in `Entity.cs`, `Component.cs`, and `Scene.cs` uses an Object-Oriented Component system which iterates through virtual update methods, causing cache misses and bad performance at scale.

**Technical Approach:**
*   **Decouple Data from Logic:** Introduce unmanaged structs for data components (e.g., `PositionComponent`, `VelocityComponent`).
*   **Entity Representation:** Entities become mere IDs (`int` or `uint`).
*   **System Implementation:** Create Systems (e.g., `MovementSystem`) that iterate over arrays of these component structs.
*   **Storage & Iteration:** Implement a lightweight Archetype or sparse-set storage for contiguous memory access to maximize CPU cache hits.
*   **API Breaking Changes:** This is a major breaking change. Existing components derived from `Component` class must be migrated to structs. The `Entity` class will likely be refactored to act as an abstraction/facade to interface with the new ECS world, keeping backward compatibility where feasible but heavily deprecating old paradigms.

### 1.2 Memory Allocations and GC in the Hot Path
**The Problem:** Core loops like `Scene.Update` and `Entity.Update` allocate memory every frame via `ToArray()` or similar list operations.

**Technical Approach:**
*   **Remove Frame-by-Frame Allocations:** Eliminate calls to `ToArray()` in update loops.
*   **Structural Change Queues:** Implement command buffers or queues for adding/removing entities and components (`_entitiesToAdd`, `_entitiesToRemove`).
*   **Deferred Execution:** Apply these structural changes strictly at the end of the frame (or beginning), ensuring the iteration over active entities/components uses existing contiguous arrays or pre-allocated lists without modification during traversal.
*   **Object Pooling:** Use object pools for any necessary runtime objects that are frequently created and destroyed to prevent GC spikes.

### 1.3 Physics Implementation (AABB vs Robust Solutions)
**The Problem:** The primitive custom AABB solver causes scaling issues ($O(N^2)$ checks) and lacks robust continuous collision detection (CCD).

**Technical Approach:**
*   **Integration of Box2D:** Replace the manual physics solver with a robust wrapper around Box2D (or a C# port like Box2D.NetStandard).
*   **Spatial Partitioning:** (If retaining custom physics) Implement a QuadTree or Spatial Hash grid to reduce collision pair checks. Box2D handles this internally (Dynamic Tree).
*   **RigidBody Overhaul:** Update the `RigidBody2D` component to interface with the Box2D simulation loop, supporting realistic impulses, forces, constraints, and CCD.

### 1.4 Rendering Architecture (System.Drawing vs Win2D)
**The Problem:** The engine currently uses CPU-bound, outdated `System.Drawing.Graphics` (GDI+) via Windows Forms instead of the hardware-accelerated `Win2D` as promised in the README.

**Technical Approach:**
*   **Win2D Integration:** Swap the `System.Drawing` backend in `Renderer2D.cs` with `Microsoft.Graphics.Canvas` (Win2D).
*   **Editor Interoperability:** Use XAML Islands (`DesktopWindowXamlSource`) to host WinUI 3 / Win2D controls within the existing Windows Forms editor shell. This bridges the gap without requiring a full rewrite of the editor UI right now.
*   **Pipeline Updates:** Refactor rendering calls (`DrawImage`, `FillRectangle`) to their Win2D equivalents, ensuring batches and hardware acceleration are properly utilized.

### 1.5 Incomplete "Completed" Phases (UI Canvas & Build System)
**The Problem:** Phases 22 (UI Canvas) and 23 (Build System) are marked as completed but lack concrete, robust implementations.

**Technical Approach:**
*   **UI Canvas Refactor:** Create a dedicated `Canvas.cs` component that acts as a root for UI elements. Implement a proper event-routing system for UI interactions (click, hover, focus) separating it from gameplay input. Render UI elements using the new Win2D backend.
*   **Build System Overhaul:** Enhance the build pipeline to generate truly deterministic packages. Integrate cross-platform MSBuild properties, define clear release/debug profiles, and ensure external assets are correctly bundled in the output directory.

## 2. Testing Strategy & CI/CD

To ensure the reliability of these sweeping architectural changes, a rigorous testing strategy is necessary.

### 2.1 Automated Unit Tests (xUnit)
*   **ECS & Data Orientation:** Write tests asserting that Systems only process Entities containing the required Component Archetypes. Validate that Entity creation and destruction are appropriately queued and resolved.
*   **Memory Benchmarking:** Implement basic performance tests or assertions that verify zero GC allocations occur during standard `UpdateFrame` ticks.
*   **Physics:** Add deterministic physics tests to verify collision resolution, impulse applications, and gravity using the new Box2D integration.

### 2.2 Editor Integration: "Run Tests" UI
*   Given the nature of the project as a comprehensive editor and engine, we will introduce a **"Run Tests"** button in the Editor UI (likely in a new "Test Runner" pane or via the Toolbar).
*   This button will programmatically invoke the `dotnet test` process for the solution or utilize a custom test runner to execute the xUnit tests directly within the editor's AppDomain.
*   Results will be parsed and displayed within the Editor's Console or a dedicated Test Results pane, providing immediate feedback during game and engine development.
